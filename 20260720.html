TL;DR

While AI agents drastically increase development velocity, they also accelerate technical debt and security risks; to achieve enterprise-grade reliability, organizations must transition from a generation-focused approach to a "Guide, Verify, Solve" framework.

The AI Productivity Paradox and "Slop"

  • The Velocity Trap: Research from Carnegie Mellon University indicates that the initial 3x to 5x boost in developer velocity provided by AI agents often dissipates within three months.
      • Technical Debt Accretion: Gains are offset by a surge in security issues, maintainability hurdles, and increased code complexity.
      • The Hallucination Problem: High-profile failures at firms like KPMG and EY demonstrate that while AI generates plausible-sounding output ("AI slop"), functional correctness does not guarantee enterprise-grade reliability.
  • Accuracy vs. Complexity: Benchmarking shows that while agents can handle complex 18-hour tasks, they currently do so at only a 50% success rate.
      • When the required accuracy rate is raised to 80%, the complexity of tasks the agents can successfully complete drops from 18 hours to approximately 3.5 hours.

The AC/DC Framework: Agent-Centric Development Cycle

  • Shift in Focus: Tariq Shaukat argues that verification must be baked into the process rather than treated as an afterthought. This is achieved through three specific disciplines:
  • Guide (Preemptive Verification): Providing agents with both "Context" and "Constraints."
      • Context: Semantic maps and architectural awareness of existing codebases (even those with billions of lines).
      • Constraints: Explicitly defining coding standards, forbidden dependencies, and "intended" (rather than just existing) architecture.
      • Efficiency Gains: Proper guidance can result in a 30% reduction in token consumption by making the "territory" easier for the agent to navigate.
  • Verify (Zero-Trust Model): Implementing a multi-layered verification fabric that assumes every model has inherent biases.
      • Algorithmic Verification: Using traditional tools to check data flows, control flows, and secrets.
      • Agentic Verification: Using AI to check for intent, business logic, and "unknown unknowns."
      • Reliability Impact: Organizations using multi-layered verification report 44% fewer AI-derived production outages.
  • Solve (Verified Maintenance): Actively using remediation agents to clean technical debt.

The Three Loops of Continuous Verification

  • The Agentic Loop: Providing in-loop verification and constraints while the agent is actively generating code to reduce errors at the source.
  • The CI Verification Loop: Combining algorithmic and agentic reviews during the pull request (PR) process to maintain high-velocity quality gates.
  • The Code Maintenance Loop: A continuous improvement process focused on keeping the codebase "clean."
      • Machine Readability: Contrary to the belief that agents don't care about code quality, clean codebases significantly reduce the reasoning, energy, and tokens required for agents to perform subsequent tasks.
      • Compounding Benefits: One large bank achieved a 92% reduction in issues by applying this three-loop approach to their agentic workflows.

TL;DR

Successful agentic AI products require a "Generative UI" layer between the model and the user to transform raw text into structured, version-aware interface components. To scale on mobile, teams must shift focus from total model latency to "time to first chunk" and use a Backend for Frontend (BFF) to ensure graceful degradation.

The Shift to Generative UI (GenUI)

  • The Problem: Model output (e.g., raw text with a phone number and hours) is often "technically correct" but requires too much manual effort from the user. Effective UX provides actionable widgets (e.g., a booking button).
  • The GenUI Spectrum: Bala Ramdoss categorizes UI generation into three levels of control:
      • Control: The model picks a pre-built component (e.g., a product card). High safety.
      • Declarative: The model composes UI from a catalog of fields and buttons (e.g., Google’s A2UI spec).
      • Open-ended: The model generates novel UI on the fly (e.g., Magic Patterns).
  • Mobile Constraints: Unlike web environments, mobile apps cannot be patched instantly. If a model sends a UI component that an older app version doesn't recognize, the app will crash. Therefore, production apps must stay within the "Control" or "Declarative" tiers.

Pattern 1: The Rendering Contract

  • Version-Aware Context: The system must maintain a repository of client capabilities. If a "Flight Card" was introduced in version 2.0, the model’s context must be engineered so it only suggests that component to users on version 2.0 or higher.
  • Typed UI Intent: The model should not return text or HTML; it should return "blocks."
      • Conversation Blocks: What the agent says.
      • UI Blocks: What the client renders.
  • Fixed Menus: In production, the model does not "invent" components. It chooses from a fixed menu provided in the prompt, ensuring the output is always something the client is programmed to handle.

Pattern 2: Optimizing for Perceived Latency

  • Time to First Chunk: Traditional metrics like "Total Latency" are less relevant for LLMs. Success is measured by how quickly the first useful UI element appears.
  • Streaming Components: Instead of waiting for a full response, the system should stream UI chunks.
      • Progressive Rendering: Show a skeleton, then partially fill it, then complete it.
      • Thinking CX: Use "thinking" indicators or progress updates (similar to Gemini) to maintain user trust during long-running tasks.
  • User Engagement: Design features like "Lens Live" that allow users to interact with the screen (tapping objects) while the model processes the final result in the background.

Pattern 3: The Role of the BFF (Backend for Frontend)

  • Server-Driven UI: The BFF acts as a buffer that absorbs model complexity, allowing the mobile client to stay "dumb and safe."
  • Hydration and Actions: The BFF does more than just format layout; it "hydrates" the UI with:
      • Action Payloads: Telling the client exactly what a tap or deep link should do.
      • Impression Metrics: Identifying which logs to fire when a component is viewed.
  • Familiarity and Branding: By using a BFF to map model intents to existing native components (e.g., a standard flight row or product card), the AI feature feels like a natural part of the brand rather than a separate, "agentic-looking" experiment.

TL;DR

Production ML security is failing not because of "exotic" AI attacks, but due to "boring" infrastructure misconfigurations—such as over-privileged accounts and flat networks—that were largely solved in traditional software years ago.

The Infrastructure Reality Gap

  • The Ray Cluster Incident: In 2023, security researchers found thousands of Ray clusters exposed on the public internet, putting over $1 billion in data at risk.
      • Root Cause: This was not a zero-day or a neural network exploit; authentication was simply "off by default," and teams failed to enable it before moving to production.
  • Legacy Playbooks for Modern Systems: While we have transitioned from deterministic 2008-era applications to probabilistic, multi-tenant ML systems, we are still using outdated security assumptions.
  • The "Boring" Failure Mode: Most breaches are caused by simple oversights: exposed API keys, over-privileged accounts, or model weights left in public storage buckets.

The Three Dominant Failure Modes

An audit of 50 real-world production ML setups found that 78% contained at least one critical security mistake. These failures consistently fall into three categories:

  • Over-Privileged Accounts: Default setups often grant full admin rights to every account with credentials that never expire.
  • Flat Networks: Systems lack internal segmentation. If an attacker gains a foothold in one minor component, they can move laterally to access the entire infrastructure.
  • Exposed Secrets and Weights: Passwords remain hardcoded in scripts, and sensitive model files are stored in unencrypted, open-access directories.

The Defense-in-Depth Framework

Securing ML requires a layered approach rather than a single "silver bullet" model defense:

  • Infrastructure (The Foundation): Focuses on container isolation, network segmentation, GPU virtualization, and encryption. If this layer fails, all higher-level controls are "just decoration."
  • Access Control: Implementing strict identity verification and the principle of least privilege.
  • Runtime Security: Monitoring live traffic for anomalies.
  • Operations and Culture: Aligning day-to-day practices, compliance, and incident response.

Security vs. Performance Trade-offs

A major reason advanced defenses fail in production is the "performance tax." Security controls must be balanced against SLA requirements for latency and throughput:

  • Basic Controls (0-8% Overhead): Logins and input checking; these are mandatory for all production systems.
  • Heavier Isolation (10-20% Overhead): Selective use for sensitive or untrusted workloads.
  • Real-time Malicious Input Detection (15-30% Overhead): While popular in research, this is often a "non-starter" for production due to high latency.
  • Maturity Levels: Organizations should aim for Level 2 (5-10% overhead) as a minimum production bar, incorporating proper logins, encryption, and network separation.

Emerging Frontiers and Future Risks

As the ecosystem matures, new threats are evolving beyond simple infrastructure mistakes:

  • Prompt Injection: The inability of models to distinguish between developer instructions and malicious user input.
  • RAG Poisoning: Malicious actors compromising the document stores that LLMs use for retrieval, thereby "poisoning" the answers without retraining the model.
  • GPU Side Channels: Risks associated with packing multiple customers on a single shared GPU, where information can potentially leak between tenants.
  • Supply Chain Vulnerabilities: The risk of downloading unverified models or add-ons from public sources.

The Evolution of Agentic Diagnostics

      • Image-Based Reasoning: A "quarantine sub-agent" converts raw metrics into image collages (similar to Grafana dashboards).

The Mandate for Velocity and Execution

  • Exponential Scale: Jeremie Eliahou Ontiveros (SemiAnalysis) highlighted that leaders like Anthropic are projected to grow from 1.5 gigawatts of capacity in 2025 to 10 gigawatts by 2027—a scale equivalent to today’s Google, but achieved in just two years.

  • Capital Velocity: CoreWeave’s open order book grew from $23 billion in March 2023 to $99 billion in March 2024, reflecting relentless global demand from model labs and hyperscalers.
  • The "Experience Gap": Successful scaling requires institutional knowledge. Richardson noted that building two data centers provides some insight, but building four or more offers an "exponential" increase in necessary operational wisdom.

Data Center and System Building Blocks

  • Vertical Integration: Vik Malyala (Supermicro) stressed the "building block" approach, where servers, networking, and liquid cooling are validated as a single unit to minimize "time to online."
  • Interoperability and Open Standards: Shaun O’Meara (Mirantis) argued for an "open API contract" between infrastructure layers. Because the hardware landscape changes every 6–18 months, infrastructure must be pluggable to avoid vendor lock-in and technical obsolescence.
  • Supply Chain and Readiness: Data center readiness is currently a major bottleneck. Over 50% of projects face delays due to power delivery issues, permitting, or the shift from traditional voltage to 400V/800V DC busbars.
  • Heterogeneous Compute: As the market matures, there is increasing demand for non-Nvidia alternatives. June Paik (FuriosaAI) noted that enterprises are seeking diversified supply chains and specialized inference chips to lower the Total Cost of Ownership (TCO).

Networking and Simulation Challenges

  • Zero-Tolerance for Error: Alex Saroyan (Netris) highlighted that networking is the "invisible" but most critical layer. Unlike traditional data centers, AI clusters have zero tolerance for configuration mistakes.
  • Digital Twin Technology: To bypass hardware lead times, operators are increasingly using digital twins to simulate and validate network architectures before the physical hardware even arrives at the data center.
  • Modern Tooling: Traditional scripting (e.g., Ansible) is often insufficient for the complexity of modern AI factories; specialized automation tools are required to manage the massive bandwidth and interconnect requirements.

The Shift to Inference and Enterprise

  • Inference Dominance: Shaun O’Meara predicted that 75% of all GPU workloads will be dedicated to inference by the end of 2027 as models move from development to production.
  • Enterprise ROI: Enterprises are moving from public cloud "sandboxes" to on-prem or private cloud deployments to control costs and data sovereignty.
  • Local Execution in Europe: Richardson noted that while the cloud is global, building is "distinctly local." Success in the European market requires navigating fragmented regulations and local power constraints rather than treating the continent as a single entity.

The Finalists: Security, Biotech, and Productivity

Three companies were selected from the top 100 to pitch live on the Master Stage, each addressing a distinct vertical of the AI revolution:

  • Curyen (Winner): A "Tech Bio" startup creating a lab-in-the-loop system for drug discovery.
      • Proprietary Data: Uses single-cell multiomics to create a "Google Maps" for human biology, making drug discovery more predictable.

  • Strawberry Browser (Second Place): An agentic browser platform designed for team collaboration.
      • Ease of Use: Positions the browser as the ideal interface for AI agents, allowing non-technical users to automate complex web tasks (e.g., CRM management and lead generation).
      • Vision: Raised $6M to build a new "agentic web" protocol that challenges the dominance of major American tech incumbents.
  • Neural Trust (Third Place): A cyber-security firm focusing on the protection of AI agents.
      • Security Layer: Offers a centralized platform (Trust Lens, Test, Gate, and Guard) to prevent agents from leaking sensitive data or deleting databases.
      • EU Endorsement: Raised a $20M seed round (the largest for a EU cyber-sec firm) and carries official investment and endorsement from the European Innovation Council.

Critical Analysis from the Judging Panel

The jury, featuring Gleb Budman (Backblaze), Cyrille Saint Olive (Google Cloud), and Luke Pappas (NEA), evaluated the startups based on "moats" and scalability.

  • The Data Moat: Judges heavily scrutinized how companies acquire and protect their data. Kinga Matuła (Curyen) emphasized an eight-year development cycle for their 1,000-step workflow as a barrier to entry.

TL;DR

Bee is a high-sensitivity AI wearable that utilizes a hardware-attested, end-to-end encrypted architecture to ensure that user data remains inaccessible to everyone—including the developers and parent company Amazon.

The Sensitivity of Continuous Audio Capture

  • Massive Data Volume: A single user typically captures approximately 10 million tokens per year through the Bee wearable.
  • Rapid Profiling: Due to the intimate nature of the device, the AI can learn virtually everything about a person—including sensitive family and financial details—within just one week of use.
  • Privacy Mission: Because Bee is "the most sensitive capture device on the market," the team’s core mission is a technical guarantee that no one at Amazon can access or look at the data.

Decentralized Key Management and Attestation

  • Device-Side Key Ownership: The encryption key is generated and managed exclusively on the user’s mobile device (iPhone or Android). It is never persisted on Bee’s backend.
  • Strict Attestation Pipeline: Before a client shares a key with the backend, it runs a sophisticated attestation process to verify the integrity of the workload.
      • Transparency Logs: Bee uses Sigstore as a public transparency log, allowing anyone to verify that the workload running in the cloud is genuine and hasn't been tampered with.
  • Memory Expiration: For keys that must reside in memory for processing, Bee enforces a seven-day expiration policy. This window was chosen to balance user utility (allowing the agent to work while the user is away from their phone) with security.

Defense Against Internal Threats at Amazon

  • The "Internal" Challenge: Standard AWS customers trust Amazon to secure their data; however, as an internal Amazon team, Bee had to build systems that defend against Amazon's own internal access.
  • Two-Tier Signing System: To prevent developers from shipping malicious code unnoticed, the system requires a separate, independent Amazon privacy team to sign off on changes.
      • Hardcoded Keys: The signing keys for this privacy team are hardcoded into the client apps and backends, making it virtually impossible for the Bee team to bypass them.
  • Auditability: The system is designed for continuous audits by third-party security firms. Because the codebase is minimized—roughly 20,000 lines of memory-safe code—it is small enough to be thoroughly verified.

Next-Generation Agent Architecture

  • Stateful Runtime: Moving beyond the "request-response" model of current LLMs, Bee utilizes a stateful runtime with persistent memory that allows the agent to work continuously on the user's behalf.
  • Autonomous Operation: The agent is fully autonomous and can connect to third-party services even when the user’s phone is offline, yet it remains under the user's encrypted control.
  • Agent Sandboxing: To prevent AI agents from "doing bad things," Bee relies on strict sandboxing rather than just prompt engineering.
      • Safety Philosophy: Korshakov compares this to the human brain's inability to stop the heart at will; the agent is physically restricted from accessing sensitive functions that could cause harm.

The Looming Grid Crisis & Data Center Demand

  • The Scale of the Problem: The U.S. grid currently generates roughly 150 gigawatts (GW) of electricity. Estimates suggest the country needs between 100 and 350 GW of new capacity by 2035.
      • Annual Growth: The grid needs to add approximately 30 GW of capacity per year over the next decade.
      • The "Lumpy" Buildout: Data center development is constrained by labor shortages and power availability. While many projects are announced, actual construction is lumpy due to permitting and interconnect delays.
  • Technological Shifts: The industry is moving toward an 800-volt architecture for new Nvidia chips, which requires more sophisticated power electronics and grid equipment.
  • Beyond Data Centers: While AI gets the headlines, the grid is under stress from "onshoring" manufacturing, the electrification of homes/vehicles, and the decommissioning of 75-year-old coal plants.

TL;DR

Agent inconsistency is a signal of ambiguity near decision boundaries rather than a simple model failure. By identifying these "flip-flops" through active learning and augmenting agents with semantic and episodic memory, developers can build trustworthy, self-evolving systems that adapt to specific customer preferences.

The Trust Problem: Why Agents Disagree

  • The "Flip-Flop" Phenomenon: Diane Lin, Lead of Self-Evolved Agents at Datadog, notes that running the same prompt through the same LLM can result in semantically different outputs (e.g., labeling a security threat as "benign" one moment and "malicious" the next).
  • Impact on Trust: In high-stakes environments like cybersecurity, inconsistency destroys product credibility. A vendor providing consistent—even if slightly less accurate—results will often win over a vendor whose model flip-flops on the same data.
  • The Gray Zone: Inconsistency typically occurs at the "decision boundary" where policies are ambiguous.
      • Human Parallel: These are often cases where even human experts would disagree because the "correct" answer depends on specific company policies or risk tolerances.
      • Information Gaps: Disagreements often signal that the agent lacks the specific context needed to disambiguate a complex scenario.

Identifying Ambiguity via Active Learning

  • Disagreement as a Signal: Rather than dismissing stochasticity as a bug, it should be used as a feature to identify "high informational" data points that require human clarification.
  • Selection Strategy: Lin recommends a "Query by Committee" or multiple-run approach.
      • Reliability: LLM uncertainty scores (probability) are often unreliable because "LLMs don't know what they don't know."
      • Disagreement Logic: Identifying where a model disagrees with itself across multiple runs is a more reliable signal that a data point is in the "gray zone" and needs intervention.
  • Efficient Human-in-the-Loop: This workflow allows teams to ignore the 75-80% of cases where the model is consistently correct, focusing human attention exclusively on the ambiguous subset where labeling provides the highest return on investment.

Memory Augmentation: Beyond Fine-Tuning

Lin proposes augmenting agents with two types of memory rather than relying on expensive, slow fine-tuning:

  • Semantic Memory (Domain Knowledge):
      • Definition: Storing factual knowledge, business policies, and explicit reasoning.
      • Application: If a customer complains about something outside a hotel's control, a semantic rule can be added to the knowledge base to consistently classify such reviews as "positive" or "ignore."
  • Episodic Memory (Past Experiences):
      • Definition: Referencing past similar cases and their resolved outcomes.
      • Application: In cybersecurity, many noise-generating alerts are recurring. Episodic memory allows the agent to see how a similar alert was handled previously and mirror that decision automatically.
  • A Complementary System: Use episodic memory to automate recurring "noisy" cases and semantic memory to distill human-reviewed "gray zone" cases into permanent policy.

Experimental Results and Takeaways

  • Cybersecurity Case Study: In a study of 93 security alerts, 25% initially showed inconsistent "flip-flop" verdicts.
      • Reduction: Applying episodic memory alone reduced inconsistency to 10%.
      • Final Resolution: The remaining 10% were resolved through human review, which then informed the semantic memory for future runs.
  • Continuous Adaptation: This workflow transforms an agent from a static tool into a "self-evolving" system that learns a customer's specific environment and preferences over time.
  • The Shift in Perspective: Stop blaming the model for inconsistency; instead, use it to find where your data or instructions lack the clarity the model needs to succeed.

This is a typical NLP task, and today’s LLMs are really good at this. However, there’s one problem. When you pipe this input to the same LLM, occasionally you will see different verdicts coming out when you run it multiple times. That means a single evaluation run didn't tell you the whole story. You need to repeat your evaluation multiple times to get a holistic picture by averaging over the results of different runs. This sounds like a technical inconvenience, but the reality is way more problematic than just an inconvenience. Let me show you an example in the cybersecurity domain. Imagine you are a security analyst where your day-to-day job is to triage security alerts and decide whether an event is malicious—meaning you need to take certain actions to stop the attack—or if it is a false alarm, meaning it is actually benign and you could ignore it. Here, the example alert you get is a failed login attempt to a Gmail account detected from a suspicious IP. If we give a task like this to an AI agent and run it multiple times, you will see that sometimes it stays consistently benign, and sometimes it stays consistently suspicious across different runs. However, there are cases where it flip-flops between benign and suspicious. Now, your customer will have difficulty using such a product because they will wonder, "Which one should I trust?" So, the inconsistency is causing a trust issue in your product. Imagine if you were in a POC bake-off; one vendor gives the consistent wording all the time, while the other vendor has this flip-flopping. You can imagine which one would win the deal. Now, I’m pretty sure you want to solve this problem. First, we have to figure out where the inconsistency comes from. The good news is that the data points that tend to flip-flop actually concentrate around the decision boundary, the so-called gray zone. Let me illustrate with the earlier example. In the sentiment analysis case, we have two reviews. The first one, no matter how many times you run it through the LLM, will always give you the positive label; in this case, it’s pretty obvious the review is super positive. However, the second case is the one that tends to flip. If we look closer, it’s kind of on the boundary. Sometimes a certain word is relatively positive, but sometimes there’s a little bit of negativity. In this case, you’ll find that even human experts will have a disagreement. In fact, there’s no right or wrong answer. It really depends on a particular company's policy and their preference. If a hotel thinks such a review is not something they can use to improve the experience, they might not label it as negative because there’s nothing they can do. But some hotels might care about it and want to label it as negative so they can improve. So, it comes down to your preference. Let’s look at the other cybersecurity example. Here is another case on the boundary. It’s on the boundary because there have been a few attempts; indeed, there’s an attacker knocking on the door, but the attacker might not have gotten in yet. For enterprise customers who always have attackers knocking on the door, if they haven't gotten into the environment, the company doesn't want to worry about it. Otherwise, they will be flooded by these types of alerts. There’s nothing they need to do because the attackers are blocked outside already. However, if the attacker does manage to get the correct password, pass MFA, and get into the environment, then it becomes very serious and you need to take certain actions. So, the behavior is almost similar at the beginning, but the outcome and the response needed are very different depending on whether it is a true attack or if the attacker is still outside. Whether we label them as malicious or benign depends on company preference regarding whether they want to be notified about such a situation. This is an example where it depends on your preference and additional information to disambiguate between the two different scenarios. From these two very different domains, you will see that the data points that tend to flip-flop are those close to the decision boundary where you need clarity to disambiguate which side it should really belong to. These data points are where human experts tend to make mistakes, and traditional machine learning classifiers will also struggle. In other words, it’s not your AI agent's fault. Your AI agent is simply pointing out the ambiguity that already exists. So now we know where the root cause of the inconsistency is. The next step is to identify where exactly they are, who they are, and how to fix them. Surprisingly—or maybe not surprisingly—the solution to identify the gray zone already exists in machine learning. It’s called active learning. The idea of active learning is that you have a lot of data points to be labeled. Imagine you have your initial model and you put it in production, maybe in monitor mode, and you want to check the quality. But you don't have the bandwidth to check all of them. Otherwise, you lose the point of having an agent do the work. But you still want to identify where the model didn't do well. Active learning is the way to select the data points where the model tends to make mistakes, and that’s what tells you where to pull your attention. Let’s go over the traditional machine learning active learning pipeline and see how it differs in the case of a new LLM or AI agent error. First, you train your initial model and then you make predictions on the unlabeled dataset, which could be the production online data. From there, you see there are certain cases where the model wasn't sure because the probability was close to 0.5. Those data points are the high-informational ones that your model is going to learn the most from. The uncertainty is a type of signal to tell you where the model tends to make a mistake. Another popular way to figure out the problematic ones is query-by-committee, meaning you use a few different models, or even the same model multiple times, and then you identify the disagreement. That’s where the model tends to make a mistake as well, and that’s where you need human clarification on the label so the model can learn from it. After you surface these problematic data points and a human takes a second look to make sure the label is actually correct—or determines if additional information is needed to disambiguate those cases—you can add those labeled data or additional features to the next round of training. Then you retrain the model and continue this cycle. You can see that in this particular active learning cycle, you spend very little effort, or at least it’s a very efficient way to identify the problematic data points, and you spend your attention on those points where your model will learn the most. That’s the traditional way. The good news is the LLM way is not very different, though there are two major parts that need some refinement: the selection strategy and the retraining. On the selection strategy, we suggest you use the one about identifying disagreement. From what we have tried so far, the uncertainty score from an LLM is not very reliable. It’s as if the LLM doesn't know what it doesn't know; when it is very confident, it doesn't mean that it is correct. However, the disagreement from different runs or from different models actually gives you a more reliable signal that the model is not sure about its verdict and needs human guidance. After you identify this group that needs attention, you label them and provide feedback like you did before. Afterward, you might be thinking, "Oh, time to retrain the model." Yes and no. Yes, one option is to retrain the model, but fine-tuning a model is expensive. Here, we are proposing a more lightweight solution that is easier to iterate: augmenting your agent with semantic and episodic memory. Let’s use the previous example to illustrate. As we talked about, in the case where the attempt flip-flops, it’s mainly missing information about a particular company's policy or preference. We can add that to the knowledge base. If a customer complains about something outside the hotel's control, then classify it as positive because you don't want to pay attention to it. In the cybersecurity example, something similar applies: you can define that a "password spray," where an attacker tries to guess your password different times continuously, should be benign if there is no successful login. In contrast, if it succeeds with a final login, then it should be malicious. You identify the additional information to help disambiguate the two and label them separately. That’s the clarity that helps your model learn and helps your human experts label more consistently. This type of domain knowledge sharpens your decision boundary, and this factual knowledge belongs to your semantic memory. This is in contrast to episodic memory. Episodic memory is for cases where you haven't gotten around to distilling the reasoning behind why an earlier case should be on the malicious or benign side. Instead, you say, "I have seen these similar cases before and it was labeled as benign," and you leverage those past similar cases to make a decision accordingly. The advantage of this episodic memory way is that it is relatively automatic. It has less human intervention; the human intervention was done in the past, and when you are doing a job online, your AI agent can reference those cases automatically. They can make decisions without waiting for you to distill the reasoning and put it into semantic knowledge. The idea is to identify a similar case to the one under examination and leverage the past decision as a reference. Imagine there’s a new case that comes in which doesn't belong to any past group; that’s the one where you still need human attention. This is also how episodic memory resolves recurring cases automatically. This is particularly useful in the case of cybersecurity alerts because a lot of the noise—especially those false positive noises—are ones that keep recurring. Those are the low-hanging fruit you can automate away. With that, your human review—which is a precious resource—can be concentrated on cases where episodic memory failed. This is also how we choose between semantic and episodic memory. The two are not contradicting; in fact, they are complementary. You use episodic memory to address recurring situations automatically. For what remains—cases still reflecting ambiguity after the reference or with no reference available—you pass them on to human review. Then, human review distills the domain knowledge and puts it into the semantic memory to be used next time. This solution brings three benefits. First, we identify data points near the decision boundary and, using domain knowledge in semantic memory and past similar cases in episodic memory, we are able to improve consistency significantly. Now you have a more trustworthy AI agent for your user. Two other byproducts are that you have very efficient quality control. You don't need to check every agent output, but instead a subset of them—the ones your active learning algorithm thinks are potentially problematic—and you provide additional information or clarify the label so your model learns. Most importantly, your labeling effort is small but has high returns. Last but not least, you are gathering customer feedback along the way. You not only have a trustworthy, consistent, and high-efficacy model, but you also have one that adapts to your customer’s environment. I’m pretty sure your customers will love your product if your AI agent listens to them and adapts to them. Here are some experimental results from real data. We collected 93 cybersecurity alerts and ran them three times. Without the solution we are proposing, a quarter of them would flip-flop their verdict. In contrast, after applying our solution using episodic memory, about 15% of those become consistent. However, there is still 10% remaining that stay inconsistent; they still flip. Sometimes that’s because there is no reference to similar past cases. Sometimes, even if there is a reference, the model still disagrees after a second thought. But that is not a problem. The episodic memory handles the automatic lockdown of the inconsistency for 15% of the cases. For the remaining 10%, a human reviews and provides additional knowledge to disambiguate those tricky cases. After that, you also adapt to the customer environment that you want to learn. The three takeaways I hope you have after this talk are: Number one, inconsistency isn't usually your model's problem. Stop blaming your model and instead focus your energy on label issues and insufficient information, and help your AI agent with this additional clarity. Second, model disagreement isn't a bug, but a feature. Treat each disagreement as an opportunity for your model to learn. Third, fine-tuning isn't your only option. Augment your agent with semantic and episodic memory. I hope you enjoy seeing your model keep improving and adapting to your customer's environment. TL;DR

AI agents fail at physical data tasks because the "physics" of large-scale unstructured data (high recompute costs and massive metadata explosion) breaks standard coding intuitions, requiring a specialized "data harness" to manage schemas, distributed execution, and persistent memory.

The "Neutron Star" Problem of Physical Data

  • Metadata Explosion: Dmitry Petrov describes physical data (video, sensors, logs) as a "neutron star"—small in file count but immense in density. For example, 2,000 video files can generate millions of individual objects, frames, and labels.
  • The Failure of Current Methods:
      • JSON-on-S3: Storing metadata in sidecar JSON files leads to extreme latency and consistency issues at scale.
      • SQL Islands: Moving metadata to a centralized database creates a "two-stack" problem, forcing researchers to toggle between Python and SQL, which slows down development.
  • Accuracy Gap: Petrov cites research from Anthropic and OpenAI showing that agents only achieve ~21% accuracy on data projects until a specific data harness or layer of context is provided.

Bridging Unstructured Data with Pydantic Schemas

  • Pythonic Data Modeling: To avoid "SQL islands," the DataChain approach uses Pydantic to define schemas. This allows developers to use the same language for data models, code, and queries.
  • Schema Transpilation: The system transpiles Python data models directly into SQL under the hood, allowing agents to query binary files as if they were rows in a structured warehouse.
  • Rich Metadata Tracking: A robust harness tracks not just the file path, but checksums (ETags), timestamps, frame IDs, and nested objects like bounding boxes, providing the agent with a transparent view of the "physical world" inside the storage bucket.

High-Efficiency Execution and Checkpointing

  • Distributed Computing: Unlike business automation where recompute is cheap, analyzing terabytes of video is expensive. A data harness must act as an execution engine (similar to Dask or Ray) that distributes Python functions across multiple nodes.
  • Incremental Updates: To prevent "losing the compute," the system must support data checkpoints. If a job fails at 90%, the harness should fix the bug and resume from the last processed file rather than restarting.
  • Avoidance of Recompute: When new files are added to a bucket, the harness should only process the "delta," updating the dataset without re-running the entire pipeline.

Testing and The Meta-Information Layer

  • The "SQL-ish" Query Shift: Petrov argues that agents should stop running expensive Python scripts on raw data to answer basic questions. Instead, they should build and query a meta-information layer.
  • Multi-Dimensional Modeling: By organizing unstructured metadata into star schemas or "one big table" formats, agents can answer complex analytical questions (e.g., "How many videos contain pedestrians?") in seconds rather than hours.
  • Agent Self-Correction: When asked a question, a well-tuned agent first asks itself: *"Do I have the proper metadata to answer this in a single query?"* If not, it builds that layer first to benefit future queries.

Memory, Lineage, and the Knowledge Base

  • Persistent Knowledge: Results should be stored in a shared Knowledge Base (organized as Markdown files) containing:
      • Session Context: Why the dataset was created.
      • Data Lineage: The connection between source files in S3, the Python code used for extraction, and the final result.
      • Dataset Stats: Previews and schemas that allow other team members (or future agent instances) to reuse the work.
  • Collaborative Efficiency: By sharing the "knowledge graph" of processed data, teams avoid paying the "double or triple price" of re-analyzing the same physical data for different projects.

The Three Pillars of Agent Failure

Most teams attempt to fix underperforming agents with larger models or more extensive RAG (Retrieval-Augmented Generation). However, Ishita Daga argues that the root causes are structural:

  • Ambiguity: Agents struggle to identify the "source of truth" among competing tables, columns, and knowledge bases.
  • Staleness: Business definitions, KPIs, and processes change rapidly, causing the agent's static context to "rot" or become deprecated.
  • Preference: Different teams often define the same metric differently (e.g., "average milestone time"), creating subjective requirements that standard models cannot navigate without specific guidance.

The Hierarchy of Truth: Solving Ambiguity

To resolve ambiguity, enterprises should organize data sources into a hierarchy, moving from the cleanest (least flexible) to the messiest (most dynamic).

  • The Semantic Layer (Tier 1): This is the highest-quality source. It contains a curated list of KPI definitions, business logic, and canonical queries. The agent should always check here first.
  • Canonical Tables & Parametric Queries (Tier 2): These provide pre-defined query structures that allow the agent some flexibility in filtering while keeping it within safe boundaries.
  • Database Graphs (Tier 3): A complex metadata layer connecting tables to specific metrics.
      • Implementation Strategy: Daga recommends starting with Tiers 1 and 2, which solve roughly 80% of problems, before attempting the high-maintenance database graph.

Managing the Context Life Cycle

To combat staleness, Daga proposes a "context life cycle" that ensures the agent’s knowledge evolves alongside the business.

  • Live Data Integration: Agents must be connected to "living" sources—such as GitHub, CRM tools, or DBT semantic layers—that are regularly reviewed and updated by human owners.
  • Event-Driven Feedback Loops: Systems should log every instance where a user identifies an incorrect definition or a new calculation method. These logs must be used to trigger context updates.
  • Rigorous Evaluation:
      • Human-Annotated Suites: Using "golden sets" of questions and answers to bench-test the agent.
      • Automated Evaluation: Comparing new agent outputs against historical "correct" answers to track performance drift.

The Subjectivity Gap: Handling Team Preferences

The most difficult challenge remains "Preference," where the "correct" answer depends entirely on who is asking the question.

  • The Definition Conflict: Team A may calculate a milestone from "completion to completion," while Team B calculates "start to start." Both are valid, but they produce different results.
  • Current Limitations:
      • Semantic Layering: Requires the user to manually prompt the model to use a specific version.
      • Agent Memory: Tools like `mem0` or `memory.md` can store preferences but often struggle to distinguish between conflicting metrics across different contexts.
  • The Goal: The industry is moving toward "Hive Mind" agents capable of routing queries to specific metrics based on the individual user’s identity or team affiliation.

The Froglet Protocol: Verifiable Workflows

  • The Verified Chain: Froglet replaces ephemeral API calls with a permanent, tamper-proof chain of signed documents: Descriptor → Offer → Quote → Deal → Receipt.
  • Identity and Security: Every actor runs a homogeneous Froglet node. Upon generation, each node creates a key pair for identity and signing. Every step of a transaction is signed on-chain to ensure the integrity of the work performed.
  • Direct Interaction: While a "Marketplace" service helps agents discover providers, the actual communication and execution happen directly between the requester and the provider without a middleman.
  • Interoperability: Froglet is designed as an interface layer. It integrates with existing payment rails, execution environments (like MCP, OpenClaw, or NemoClaw), and network protocols rather than replacing them.

  • Two-Part Payment Model: To balance risk, Froglet utilizes a specific payment structure:

      • Success Fee: Protects requesters from malicious or underperforming providers by only triggering upon successful delivery.
  • Agentic Commerce: The goal is to move away from traditional checkout pages toward verifiable work. This allows receipts, identities, and workload hashes to survive across different models, hosts, and marketplaces.
