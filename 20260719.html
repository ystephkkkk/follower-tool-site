TL;DR: Effective agent evaluation requires moving beyond generic "helpful assistant" metrics toward adversarial user simulators fine-tuned on real production data and actionable, binary rubrics calibrated against expert human labels.

The Problem with Synthetic Evaluations

  • The "Polite LLM" Trap: Off-the-shelf frontier models are trained to be helpful assistants. When used to simulate users in offline tests, they are too patient and verbose, leading to inflated "pass rates" (e.g., 90% accuracy) that collapse in production.
  • Production Reality: Real customers are often frustrated, impatient, or skeptical. They use messy language and have diverse personas that generic synthetic datasets fail to capture.
  • Non-Actionable Metrics: Many teams rely on generic scores (e.g., "helpfulness: 0.7"). These scores are floating values that don't provide a clear signal on whether a model is ready for launch or how to fix a failure.

High-Fidelity Adversarial Simulation

  • Fine-Tuning on Transcripts: To solve the "polite LLM" problem, Lyft fine-tuned an LLM on real rider and driver transcripts. This creates a simulator that mimics the actual distribution, tone, and frustration levels of real users.
  • Persona-Based Testing: Simulations are grounded in specific user personas, such as:
      • Refund Seekers: Users primarily motivated by compensation.
      • AI Skeptics: Users who immediately demand a human agent.
      • Bypassers: Users who ignore AI instructions to reach a specific outcome.
  • The World State: Simulations define a "World State" for the user (e.g., a luxury driver with a 5-year history) to ensure the AI agent is tested against specific, data-rich scenarios rather than vague queries.

Building and Calibrating the "LLM Judge"

  • Binary Rubrics: Instead of subjective 0-1 scores, use binary (pass/fail) outcomes tied to specific business goals. This makes it easier to calibrate the judge and results in more consistent scoring.
  • The Education Rubric: A specific metric used at Lyft to see if an agent is "over-educating" the user or escalating to a human too quickly.
  • Judge as Classifier: Treat the LLM Judge like a traditional machine learning classifier.
      • Labeling: Hand-label ~100 examples to create a ground-truth "Gold Set."
      • Validation: Measure the judge’s performance using precision and recall against the human labels.
      • Iterative Prompting: Use a "Train/Dev/Test" split of the labeled data to refine the judge's prompt without over-fitting to specific examples.
  • Statistical Rigor: Avoid making shipping decisions based on small point-estimate gains. Every score should include a confidence interval to ensure that a 4% improvement in a small sample size isn't just statistical noise.

The Evaluation Lifecycle & Harness

  • Trace Logging: Use tools like LangSmith or LangFuse to capture full graph executions, including node transitions, tool calls, and token usage.
  • Annotation Cues: Provide domain experts with a simplified UI to label traces, which feeds back into the evaluation dataset and helps identify "criteria drift" (when the definition of a "good" response changes over time).
  • The Eval Harness: A standardized, config-driven (YAML) system that allows any engineer or analyst to run benchmarks.
      • Pre-commit Hooks: Running evals locally to catch regressions before code is pushed.
      • CI/CD Integration: Using the harness as a formal "Launch Gate" for production deployments.
  • Continuous Learning: Close the loop by using failure modes to update the system prompt (Harness Learning), the RAG knowledge base (Context Learning), or the underlying model weights (Model Learning).

TL;DR

May Walter (CTO at Hud) details an agentic workflow that automates the unpredictable "research phase" of performance optimization, using production runtime intelligence to identify, fix, and verify high-ROI code improvements.

The Problem: The Unpredictable "Research Black Box"

  • Investigation vs. Execution: Teams often avoid performance optimizations because while the fix might take an hour, the investigation could take weeks.
  • The "Leaky Bucket" Cycle: Performance issues are ignored until they reach a crisis mode; teams fix them in an emergency, then immediately return to ignoring the debt because proactive monitoring is too expensive.
  • Institutional Knowledge Gaps: Mature codebases often rely on a single "Dave" who understands the legacy code, making performance tasks risky for anyone else to touch.

Bridging the "Prod-to-Code" Context Gap

  • The Data Mismatch: Standard metrics (CPU, memory, P90 latencies) exist at the service/endpoint level, but coding agents reason at the function/file level.
  • Functional Runtime Intelligence: Hud captures "forensic context" that connects production traces directly to specific lines of code.
      • Forensic Evidence: The system captures deep data (queries, latencies, outbound calls) only when a request exceeds a threshold (e.g., P99), preventing data noise.
      • Heads-Up Display (HUD): A data structure that presents production behavior in a format comprehensive to an engineer or agent.
  • Agentic Skills & MCP: Using the Model Context Protocol (MCP) and a set of "skills," agents can query ClickHouse data to understand memory spikes or 500-level errors by comparing current behavior against a baseline.

Technical Guardrails for Autonomous Optimization

  • Overcoming LLM Failure Modes:
      • Plausible Unverified Fixes: Agents often suggest "correct-looking" fixes that don't actually solve the production bottleneck. Verification in a test environment is mandatory.
      • The "Lazy Fix": Agents may try to simply catch an exception rather than solving the root cause of why a result is lagging.
      • Complex Query Handling: Special logic was required to help agents handle columnar databases like ClickHouse, which differ from standard SQL patterns.
  • Verification Loop: The agent identifies the root cause, writes the fix, reruns tests, and measures the impact on the specific flow before ever involving a human.

Operationalizing Agentic PRs

  • Avoiding "PR Rain": Flooding developers with 80 automated PRs causes friction. Instead, the system generates a "human-friendly report" in Slack or Teams.
  • High-ROI Scoring: Opportunities are ranked based on:
      • Impact: How often the code path runs and its criticality to business (e.g., payment flows vs. internal tools).
      • Risk: The complexity of the change and whether it requires database migrations.
  • The "Gist" Approach: Reports provide a short explanation of the problem (e.g., "Using distinct instead of a Mongo search index") and the proposed fix, allowing the human to act as a high-level "review gate."

TL;DR

Modern software architecture is built on the assumption of deterministic human or API interactions; however, AI agents are non-deterministic and operate at scales that break these legacy identity and authorization models, requiring a shift toward context-aware, fine-grained, and "just-in-time" permissioning.

The Failure of Human-Centric Architectural Assumptions

  • Scaling Bottlenecks: Traditional "human-scale" features can become unintended DDoS attacks when accessed by agents.
      • The "Last Seen" Example: Scalekit experienced rhythmic latency spikes because a simple "last active" timestamp—designed for humans logging in occasionally—was being updated 60 times per second by an agent.
  • Non-Deterministic Behavior: Unlike traditional software scripts, agents are probabilistic.
      • Code vs. LLM: Human-written programs are deterministic and stay in their "lane." Agents can behave differently from one run to the next, making it impossible to rely on traditional code reviews for security guarantees.
  • The Permission Gap: Developers currently grant agents broad "default" permissions because existing primitives do not allow for the necessary level of granularity.

Limitations of Legacy Authorization Models

  • Identity Confusion: Current systems generally use two slots: the Human User (Principal = Actor) or the Service Account (Machine Identity). Neither accounts for an agent acting on behalf of a specific user.
  • Broad OAuth Scopes: Standard OAuth scopes are too coarse for AI.
      • The Gmail Problem: A scope like "send emails" is dangerous for an agent. Current systems lack the "attribute-level" scoping required to limit an agent to specific recipients, senders, or time windows.
  • The MCP Disconnect: Many Model Context Protocol (MCP) servers surface all available tools to an agent regardless of the specific user's authorization level, leading to "hallucinated" tool calls or unauthorized actions.

Engineering Deterministic Guardrails

  • Identity Binding: The agent must have its own distinct identity while remaining strictly bound to the principal (the user) it is representing at all times.
  • Fine-Grained Scoping: Architecture must move beyond broad permissions toward:
      • Attribute-Level Scoping: Restricting access based on specific data fields.
      • Context-Level Scoping: Limiting actions based on the specific goal or environment.
      • Temporal Access: Providing elevated permissions only for the duration of a specific task.
  • Just-in-Time (JIT) Authorization: Rather than static permissions granted at registration, agents should operate under "least privilege" by default and request elevated permissions dynamically as the workflow requires.

Visibility and Strategy

  • Auditability Requirements: Systems must maintain absolute visibility into every action, identifying who authorized it, what specific context was provided, and how long the authorization lasted.
  • Operational Strategy: Relying on the hope that an agent won't "go rogue" is a liability.
      • Proactive Defense: Organizations must implement deterministic authorization controls that function as hard boundaries the agent physically cannot cross, regardless of its prompt or model logic.

TL;DR

Foundation models for single-cell biology aim to accelerate drug discovery and enable "cellular rejuvenation," though current transformer-based architectures often struggle with the inherent noise and high dimensionality of biological data compared to newer flow matching approaches.

The Strategic Importance of Single-Cell Biology

  • Cellular Rejuvenation: Altos Labs focuses on restoring cell health to reverse disease. This is inspired by the Yamanaka factors, four transcription factors that can reprogram old skin cells back into young embryonic stem cells.
  • The "Digital Twin" Vision: The ultimate goal is to move from modeling single cells to tissues, organs, and eventually a "virtual human." This could drastically improve our understanding of disease and drug response.
  • Counteracting Eroom’s Law: While computing power follows Moore's Law, drug development efficiency is declining. Developing a single drug takes roughly 10 years and billions of dollars with a 95% failure rate; AI foundation models aim to compress this timeline and increase success rates.

Measuring the Single Cell: Data Modalities and Scale

  • RNA-seq (Transcriptomics): This is the primary modality for foundation models. It measures which of the ~20,000 genes are expressed in a cell and in what quantity.
  • Data Availability: Current datasets range from tens of millions to nearly one billion cells. While RNA-seq is high-throughput and scalable, other modalities like Proteomics (measuring proteins) remain difficult to scale due to low throughput.
  • The "Snapshot" Problem: Biology is a continuous movie of bursts and cycles, but current sequencing technology only captures a static snapshot. This makes it difficult to model temporal changes or cell trajectories accurately.
  • Heterogeneity and Noise: Even identical cells may not "read" the same way due to biological variance and technical noise introduced by different laboratory machines.

Engineering Challenges for Foundation Models

  • Transformer-Based Architectures: Models like scGPT and Informer treat cells as sentences and genes as tokens. They use BERT-style masking to predict gene counts and understand gene-to-gene relationships.
  • The Compression Dilemma: Many current models compress single-cell data into a latent vector. Akram Baharlouei notes that this often results in significant information loss, sometimes leading to simple linear models outperforming complex transformers.
  • Benchmarking Realities: Comprehensive benchmarking on tasks like perturbation response modeling shows that expensive foundation models often underperform in biology compared to their counterparts in the language and imaging domains.

The Shift to Flow Matching

  • PrimeFlow: Newer research suggests that flow matching models, such as PrimeFlow, are more effective than autoregressive models for biological data.
  • Distribution Matching: Unlike autoencoders that try to predict the "mean" of the data, flow matching models attempt to match the actual distribution of the data, starting from Gaussian noise.
  • Scalability Requirements: To achieve the "GPT moment" for biology, the field requires massive scaling not just in data quantity, but in the quality and realism of how that data is measured.
